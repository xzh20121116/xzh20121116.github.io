分布式锁

阅读量: 今晚打老虎 2020-05-31 20:43:04

1. zk是啥？

数据库、文件存储系统，并且有监听通知机制（观察者模式）

2. 特性

  1. 有序节点

    假如 当前有一个父节点为 /lock，我们可以在这个父节点下面创建子节点，使用有序节点，我创建子节点 /lock/node- ，并且指明有序，那么 zookeeper 在生成子节点的时候会自动的添加整数序号

  2. 临时节点

    客户端可以建立临时节点，回话超时，或者客户端断开连接之后，zk 会自动删除该节点

  3. 时间监听

    在读取数据时，可以同时对节点设置事件监听，当节点数据或结构发生变化是，zk 会通知客户端

    四种事件

    1. 节点创建
    2. 节点删除
    3. 节点数据修改
    4. 子节点变更

3. 应用场景

  • 服务注册与订阅（共用节点）
  • 分布式通知(监听 znode)
  • 服务命名（znode 特性）
  • 数据订阅、发布（watcher）
  • 分布式锁（临时节点）

4. 文件系统存了什么？

节点，节点名称都是唯一的

zk 的节点有四大类型

  • 持久化节点（zk 断开节点还在）

  • 持久化顺序编号目录节点

  • 临时目录节点（客户端断开连接后节点就删除了）

  • 临时目录编号目录节点

5. 节点怎么创建？

5.1. 创建永久节点

create /test node

5.2. 创建临时节点

create -e /linshi node

如果断开这次连接，这个节点就消失了

5.3. 创建顺序节点

create -s /shunxu node

5.4. 临时顺序节点

create -e -s /linshishunxu  node

6. 分布式锁

6.1. 算法流程–未优化版

  1. 客户端连接 zk，并在/lock 下创建 临时有序节点，第一个为/lock/lock-0000000000，第二个为/lock/lock-0000000001，以此类推。

  2. 客户端获取 /lock下的子节点列表，判断自己创建的节点是否为当前子节点列表中序号最小的子节点

    • 序号最好—则认为获得锁
    • 非最小—监听/lock的子节点变更消息，获得变更通知后重复此步骤，直到获得锁

  3. 获得锁的执行业务代码

  4. 完成业务代码，删除对应的子节点释放锁

6.2. 为什么用临时节点

临时节点保证客户端获取锁后，宕机了，zk 会自动删除这个临时节点，相当于自动释放锁了

6.3. 在【算法流程–未优化版】的第二步获取列表与监听不是原子操作？

这个问题不会出现，下面的流程是不会发生的

客户端 A 创建序号 1 的节点，客户点 B 创建序号 2 的节点。

  1. A 获取 列表，发现自己最小，开始执行业务代码
  2. B 获取列表，发现自己不是最小，准备设置监听（此时b让出了 cpu 时间，并未设置）
  3. A 执行完毕，删出列表
  4. B 设置监听（因为 A 在第3 步就删除了 ，此时 B 监听不到变化了，永远等待？）

因为 zk 提供的 API 中设置监听器的操作和读操作是原子操作，为的就是保证不丢失事件

6.4. 算法流程–终极

6.4.1. 为什么需要优化

假如当前有 1000 个节点在等待锁，如果获得锁的客户端释放锁，那么这 1000 个客户端都会被唤醒，

这被称为羊群效应，在这种羊群效应中，zk 需要通知 1000 个客户端，会阻塞其他操作

最好的情况是，只唤醒新的最小节点对应的客户端

6.4.2. 应该怎么做呢？

在设置监听事件时，每个客户端应该监听他前面的一个节点，2 监听 1，1 监听 0

6.4.3. 算法流程–终极版

  1. 客户端连接 zk，并在/lock 下创建 临时有序节点，第一个为/lock/lock-0000000000，第二个为/lock/lock-0000000001，以此类推。
  2. 客户端获取 /lock下的子节点列表，判断自己创建的节点是否为当前子节点列表中序号最小的子节点
    • 序号最好—则认为获得锁
    • 非最小—监听自己前面的子节点变更消息，获得变更通知后重复此步骤，直到获得锁
  3. 获得锁的执行业务代码
  4. 完成业务代码，删除对应的子节点释放锁
